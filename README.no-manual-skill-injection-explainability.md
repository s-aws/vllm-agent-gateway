# No Manual Skill Injection Explainability

Phase 204 proves that founder-facing natural-language prompts can select workflows, skills, and tools without the user naming skill IDs, pasting `SKILL.md`, or referencing internal workflow envelopes.

The gate adapts the governed founder-field prompt catalog to the existing selector explainability validator. It validates the live workflow-router artifacts generated by gateway and AnythingLLM runs; it does not add a second selector or formatter.

## What It Enforces

- Prompt text does not contain internal workflow IDs, route-decision terms, or skill IDs from the matrix.
- Each prompt maps to the Phase 203 workflow/skill/tool selection matrix.
- Chat output includes selected workflow, selected skills, selected tools, route rules, rejected-candidate counts, and grounding markers.
- Route artifacts agree with the chat-visible selector explanation.
- Selection policy remains metadata-only, fail-closed on low confidence, and does not require manual skill injection.
- Protected frozen fixtures remain unchanged.

## Commands

Offline policy and prompt-contract validation:

```bash
python3 scripts/validate_no_manual_skill_injection_explainability.py
```

Live gateway and AnythingLLM validation:

```bash
python3 scripts/validate_no_manual_skill_injection_explainability.py --live --timeout-seconds 900
```

Expected passing marker:

```text
PHASE204 NO MANUAL SKILL INJECTION EXPLAINABILITY PASS
```

## Boundary

Phase 204 is an M3 selection explainability gate. It proves natural prompt selection and visible selector reasoning; Phase 205 still must replay target and holdout prompts for route stability.

Offline mode is a preflight only and is not phase-closeout proof. Phase closeout requires `--live` without `--allow-partial`.

For debugging only, filtered runs may use `--case-id <id> --allow-partial`. Do not use partial mode for phase closeout.

Approval-gated disposable apply prompts are intentionally excluded from this Phase 204 closeout while the active model-capability profile blocks that task class without explicit approval. Those prompts remain covered by approval/apply validation phases.
