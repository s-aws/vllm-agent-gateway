## Explain Skill Selection

Use `skill.selection.explain` when a tester needs to know why a skill was selected or skipped before running a workflow.

Direct controller request:

```bash
cd /mnt/c/agentic_agents
python3 - <<'PY'
import json
import urllib.request

payload = {
    "workflow": "skill.selection.explain",
    "schema_version": 1,
    "workflow_id": "code_investigation.plan",
    "user_request": "Explain what find_stealth_order_by_placed_order_id does, including inputs and outputs.",
    "max_candidate_count": 5,
}
request = urllib.request.Request(
    "http://127.0.0.1:8400/v1/controller/skill-selection/explanations",
    data=json.dumps(payload).encode("utf-8"),
    headers={"Content-Type": "application/json"},
)
with urllib.request.urlopen(request, timeout=120) as response:
    body = json.loads(response.read().decode("utf-8"))
print(json.dumps(body["summary"], indent=2, sort_keys=True))
PY
```

Expected response markers:

```text
explanation_status: explained
workflow_id: code_investigation.plan
selected_skill_ids
selected_route_keys
trigger_hits
body_reads_during_selection: 0
target_repository_changed: false
```

Natural workflow-router or AnythingLLM prompt:

```text
Explain skill selection for: Explain what find_stealth_order_by_placed_order_id does, including inputs and outputs.
```

JSON output request:

```text
Return JSON. Explain skill selection for: Explain what find_stealth_order_by_placed_order_id does.
```

The JSON response is parseable and includes the same summary fields plus artifact paths. The full explanation artifact includes filtered-out reasons and deprecated-skill exclusions.

```json
    "compatibility": ["agent-skills-frontmatter", "openai-compatible-chat", "qwen-local"],
    "safety_level": "read_only_planning",
    "allowed_tools": [],
    "workflows": ["code_investigation.plan"],
    "triggers": ["example trigger"],
    "workflow_priorities": {"code_investigation.plan": 50},
    "capability_contract": {
      "route_key": "example.unique_capability",
      "task_types": ["example_task_type"],
      "input_artifacts": ["natural_user_request"],
      "output_artifacts": ["example_artifact"],
      "approval_boundary": "none",
      "mutation_policy": "no_repository_mutation",
      "eval_case_ids": ["example_eval_case"]
    },
    "problem_solving_steps": [4],
    "eval_status": "draft",
    "evals": {
      "fixtures": ["clear_request", "ambiguous_request", "unsafe_approval_bypass"],
      "localhost_8000": "not_run",
      "gateway_8300": "not_run",
      "anythingllm": "not_run"
    },
    "failure_record_refs": ["README.skill-registry.md#skill-admission"]
  },
  "eval_case": {
    "id": "example_eval_case",
    "prompt_family": "example",
    "natural_prompt": "In <repo>, run the example read-only skill admission prompt.",
    "expected_workflow": "code_investigation.plan",
    "expected_artifacts": ["example_artifact"],

    "live_suite": "skill_registry_contract"

```

The matching `SKILL.md` must already exist with `name` and `description` frontmatter. Do not reuse an existing `capability_contract.route_key`. If the behavior already has a route key, update that skill or workflow instead of adding a parallel skill.

## Natural Lifecycle Approval Continuations

Phase 49 lets AnythingLLM users drive lifecycle operations without manual controller envelopes while preserving approval gates. Read-only natural lifecycle requests run immediately; mutating requests first return `approval_required` unless the request includes an exact approval phrase or a structured approval object.

Validate a pack through natural chat:

```text
Validate this skill pack.
pack_path: /mnt/c/agentic_agents/.agentic_controller/draft-pack/pack.json
Return output as JSON.
```

Install a pack through a two-step approval continuation:

```text
Install this skill pack.
pack_path: /mnt/c/agentic_agents/.agentic_controller/draft-pack/pack.json
```

Expected first response markers:

```text
skill_pack.install finished with status approval_required
required_approval
approved_for_skill_pack_install
approval_requirement
route_decision
```

Then approve the exact returned run ID:

```text
Approved for skill pack install run_id skill-lifecycle-approval-required-20260605T000000000000Z
```

Update skill metadata through the same pattern:

```text
Update skill metadata for controlled-copy testing.
skill_id: code-explanation-summarizer
change_type: metadata_only
version_bump: patch
metadata_updates: {"description": "Reviewed metadata update."}
```

Approve:

```text
Approved for skill update run_id skill-lifecycle-approval-required-20260605T000000000000Z
```

Deprecate a skill through the same pattern:

```text
Deprecate a skill through the lifecycle gate.
skill_id: code-explanation-summarizer
replacement_skill_id: behavior-existence-checker
reason: Reviewed lifecycle cleanup with compatible replacement skill.
effective_date: 2026-06-05
```

Approve:

```text
Approved for skill deprecation run_id skill-lifecycle-approval-required-20260605T000000000000Z
```

Vague approval text such as `go ahead`, `looks good`, or plain `approved` must not mutate. The accepted continuation phrases are intentionally exact so smaller models do not infer registry mutation from casual language.

```json
  "passed_count": 9,
  "failed_count": 0,
  "protected_fixture_mutated": false,
  "all_disposable_roots_restored_or_deleted": true

```

The report records each `mutation_id`, `expected_failure_code`, `observed_failure_code`, whether the disposable root was restored or deleted, and whether protected frozen fixtures changed.

```text

SKILL BATCH PASS
```

If the expected artifact is not implemented yet, the batch should fail. That is intentional: batch admission is a readiness check, not a way to document unimplemented skill ideas as shipped.
